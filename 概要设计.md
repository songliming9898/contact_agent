# 软件公司合同问数 Agent — 概要设计文档

## 1. 项目概述

### 1.1 项目名称
ContractAgent — 软件公司合同智能问数系统

### 1.2 项目目标
基于 **LangChain Agent 框架**，构建双 Agent 协作的合同智能系统：

- **合同解析 Agent**：自动解析 Word/PDF 合同文档，通过 LLM 提取关键字段，输出标准化 JSON 结构化数据
- **智能问数 Agent**：基于 **MySQL 结构化数据 + ChromaDB 向量检索**，支持自然语言问答

用户打开手机页面直接进入对话，无需登录。后台管理页使用固定密码登录，支持批量导入合同。

### 1.3 部署环境
- **操作系统**：Alibaba Cloud Linux 3.2104 U12 (OpenAnolis Edition)
- **Python 版本**：Python 3.9+
- **硬件要求**：2C4G 起步（Demo 阶段）
- **访问方式**：手机浏览器（微信/企业微信内置浏览器兼容）

### 2.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **Agent 框架** | **LangChain (ReAct + Tool)** | **业界标准 Agent 框架，内置工具调用链** |
| Web 框架 | FastAPI | 高性能异步框架，原生支持 OpenAPI |
| **前端 UI** | **Vue3 + Vant4** | **移动端优先 UI 组件库，适配手机** |
| 前端构建 | Vite | 快速构建，HMR 热更新 |
| 后台认证 | 固定密码（环境变量配置） | Demo 极简方案 |
| 文档读取 | python-docx + PyMuPDF | Word/PDF 文本提取 |
| 合同解析 | LLM Function Calling | 用 LLM 提取字段→标准化 JSON |
| **结构化存储** | **MySQL 8.0** | **金额汇总/统计/多维筛选/精确查询** |
| MySQL 驱动 | PyMySQL + DBUtils | 纯 Python，连接池管理 |
| **文本分块** | **LangChain TextSplitter** | **按合同条款层级语义切分** |
| **Embedding** | **BAAI/bge-base-zh-v1.5** | **768维，中文法律/合同语义效果好** |
| Embedding 备选 | BAAI/bge-large-zh-v1.5 | 1024维，精度更高（生产推荐） |
| **向量数据库** | **ChromaDB** | **轻量级本地部署，零配置** |
| LLM | 阿里云 DashScope (通义千问) | 中文能力强，API 调用方便 |
| LLM 备选 | 本地 Ollama (qwen2:7b) | 离线可用，Demo 够用 |
| 流式输出 | SSE (Server-Sent Events) | 手机端打字机效果，体验好 |

> **双存储设计理念**：
> - **MySQL** 负责结构化数据的精确查询：金额汇总、合同统计、按甲乙方筛选、分组统计等
> - **ChromaDB** 负责非结构化文本的语义检索：条款原文搜索、模糊匹配、跨字段语义理解
> - 两个存储互为补充，Agent 根据问题类型自动选择合适的工具

### 3.2 文档读取模块 (`document_reader.py`)

**功能**：读取 Word(.docx) 和 PDF 文件的原始文本。

```
输入：原始文件路径
输出：{ "filename": "xxx", "full_text": "...", "page_count": N }
```

**处理流程**：
1. 根据文件后缀 `.docx` / `.pdf` 分发读取器
2. Word：python-docx 逐段提取，保留段落结构
3. PDF：PyMuPDF (fitz) 逐页提取
4. 基础清洗（去除乱码、不可见字符，保留原始换行结构）

### 3.3 合同解析 Agent (`contract_parser_agent.py`)

**功能**：将合同原始文本通过 LLM 提取为标准化 JSON。

**Agent 类型**：LangChain Chain（非 ReAct，是固定流水线）

**处理流程**：
```
原始文本
  → 文本预处理（过长则分段）
  → LLM Function Calling（定义 JSON Schema 输出）
  → JSON 校验与修复
  → 返回标准化 contract_json（不再存本地 JSON 文件）
```

**LLM Prompt 设计**：

```
你是一个专业的合同解析专家。请从以下软件合同文本中提取关键信息，
严格按照指定的 JSON Schema 返回结果。

合同文本：
{contract_text}

要求：
1. 合同金额统一为人民币"元"单位
2. 日期格式为 YYYY-MM-DD
3. 如果合同中没有某个字段，填 null
4. 付款方式包括：一次性付款/分期付款/按里程碑付款
5. 识别尾款金额和尾款支付条件
6. 提取试用期条款（如有）
7. 提取售后服务条款（如有）
```

### 3.4 智能问数 Agent (`contract_query_agent.py`)

**功能**：基于 LangChain ReAct Agent，使用 8 种 Tool（5个MySQL查询 + 1个ChromaDB + 2个MySQL写）回答用户问题。

**Agent 类型**：LangChain `create_react_agent` + `AgentExecutor`

**工具3层架构设计**：

```
┌──────────────────────────────────────────────────────┐
│ Layer 1: 大模型层 (tools.py)                          │
│ 5个 MySQL查询 + 1个 ChromaDB + 2个MySQL写 @tool        │
│ 直接暴露给 LLM，描述清晰，参数简单                      │
├──────────────────────────────────────────────────────┤
│ Layer 2: 工具路由层 (tool_router.py)                   │
│ 格式化/聚合/路由，把数据执行层结果适配为大模型友好格式     │
├──────────────────────────────────────────────────────┤
│ Layer 3: 数据执行层 (data_layer.py)                    │
│ DAO 操作，直接执行 SQL / ChromaDB 查询                 │
│ 纯数据操作，不关心上层格式                              │
└──────────────────────────────────────────────────────┘
```

**ReAct 推理流程**：
```
用户: "保密协议的保密期限是多久？"

Agent Thought: 用户问保密期限，需要从合同中查找。
               先用结构化查询看有没有直接数据。

Action: get_con_details("保密协议")
Observation: [{"contract_id":"NDA-001","after_sales_service":null,...}]
             结构化字段没有直接命中保密期限

Agent Thought: 结构化数据未覆盖保密期限字段，
               需要向量检索找相关条款原文。

Action: search_vector("保密协议的保密期限")
Observation: ["第5条 保密期限：本协议保密期限为合同终止后5年...",
              "第6条 违约责任..."]

Agent Thought: 找到了保密期限为5年，综合回答。

Final Answer: 根据保密协议（NDA-001）第5条，
             保密期限为合同终止后5年。
```

**公司简称模糊匹配示例**：

```
用户: "华为公司的合同总金额是多少？"

Agent Thought: 用户提到"华为公司"，数据库里存的是完整公司名
              （如"华为技术有限公司"），需要先用模糊搜索定位。

Action: search_con_text("华为")
        → L3 dao_search_text 对 party_a/party_b 执行 LIKE '%华为%'
        → 同时返回匹配到的甲乙方列表和金额摘要

Observation: {
  "matched_parties": [
    {"party_a":"华为技术有限公司","party_b":"XX软件","total":500000},
    {"party_a":"YY科技","party_b":"华为终端有限公司","total":800000},
    {"party_a":"深圳华为云计算技术有限公司","party_b":"ZZ信息","total":300000}
  ],
  "total_amount": 1600000.00,
  "contract_count": 3
}

Agent Thought: 搜索到3份与"华为"相关的合同。
              涉及3个不同的华为实体公司。
              总金额160万已由search_con_text直接返回。

Final Answer: 与华为相关的合同共3份，总金额¥1,600,000.00：
             ① 华为技术有限公司 × XX软件 — ¥500,000
             ② YY科技 × 华为终端有限公司 — ¥800,000
             ③ 深圳华为云计算技术有限公司 × ZZ信息 — ¥300,000
```

> **核心设计**：`search_con_text` 不仅搜索文本内容，还**专门针对 `party_a` 和 `party_b` 做 LIKE 模糊匹配**，一步返回匹配到的公司全称列表 + 金额汇总。Agent 不需要额外调用 `query_con_sum`，减少工具调用次数。

**Agent Tools（8个工具 — 5个查询MySQL + 1个查询ChromaDB + 2个写MySQL）**：

| 层级 | Tool 名称 | 功能 | 数据源 | 类型 |
|------|-----------|------|--------|------|
| L1 | `query_con_sum` | 金额汇总查询（总金额/尾款总额/平均金额等） | MySQL | 读 |
| L1 | `query_con_count` | 合同数量统计（按甲乙方/年份/类型分组） | MySQL | 读 |
| L1 | `search_con_text` | 关键词模糊搜索（甲乙方名称+合同全文），支持简称匹配完整公司名，同时返回匹配到的金额汇总 | MySQL | 读 |
| L1 | `get_con_details` | 获取单份合同完整详情 | MySQL | 读 |
| L1 | `list_contracts` | 分页列出合同概要信息 | MySQL | 读 |
| L1 | `search_vector` | 语义向量检索原文条款片段 | ChromaDB | 读 |
| L1 | `insert_contract` | 插入新合同（结构化JSON→MySQL三写） | MySQL | 写 |
| L1 | `delete_contract` | 删除合同（MySQL级联删除+ChromaDB同步清理） | MySQL+ChromaDB | 写 |

> **写工具说明**：
> - `insert_contract`：后台管理批量导入时，每份合同解析完成后调用，执行 MySQL INSERT（contracts + installments + products + key_clauses）
> - `delete_contract`：后台管理删除合同时调用，执行 MySQL DELETE CASCADE + ChromaDB 对应向量数据删除
>
> **公司简称模糊匹配策略（`search_con_text`）**：
> 用户常用简称（如"华为""腾讯"）提问，数据库存的是完整公司名。`search_con_text` 内置模糊匹配逻辑：
> ```
> 输入: keyword = "华为"
> ↓
> L3 dao_search_text 执行:
>   1. 甲乙方模糊搜索:
>      SELECT contract_id, party_a, party_b, project_name, total_amount
>      FROM contracts
>      WHERE party_a LIKE '%华为%' OR party_b LIKE '%华为%'
>   2. 合同全文搜索:
>      SELECT contract_id, party_a, party_b, SUBSTRING(full_text, ...) as snippet
>      FROM contracts
>      WHERE full_text LIKE '%华为%'
>   3. 合并去重，返回:
>      { matched_parties: [...], snippets: [...], total_amount: N, count: N }
> ↓
> Agent 直接获得匹配到的完整公司名 + 金额，无需额外工具调用
> ```
> 如果 `search_con_text` 无结果（如用户说"鹅厂"→"腾讯"），Agent 的 LLM 知识可推理出对应公司名，重试搜索。

**工具路由层关键设计**：

```python
# tool_router.py — Layer 2

    """金额汇总路由：调用 data_layer 查询，格式化返回"""
    result = dao_query_amount(field, filters)
    return f"合同总额: ¥{result['total']:,.2f}\n合同数量: {result['count']}"

def route_search_con_text(keyword: str, limit: int = 10) -> str:
    """
    关键词模糊搜索路由（含公司简称匹配）：
    1. 调用 dao_search_text 对 party_a/party_b 做 LIKE '%keyword%' 匹配
    2. 同时搜索 full_text 全文
    3. 合并结果，返回匹配到的公司全称 + 金额汇总 + 文本片段
    """
    results = dao_search_text(keyword, limit)
    # 格式化为 LLM 友好文本，突出显示匹配到的完整公司名
    lines = []
    if results["matched_parties"]:
        lines.append(f"匹配到 {results['count']} 份合同，总金额 ¥{results['total_amount']:,.2f}:")
        for r in results["matched_parties"]:
            lines.append(f"  • {r['party_a']} ↔ {r['party_b']} | ¥{r['total']:,.2f}")
    if results["snippets"]:
        lines.append("\n相关原文片段:")
        for s in results["snippets"]:
            lines.append(f"  [{s['contract_id']}] ...{s['snippet']}...")
    return "\n".join(lines)
```

### 3.6 合同文档专用 Embedding 策略 (`embedding_pipeline.py`)

> **核心原则**：合同文档不是普通文档，必须针对合同语义特点做专项 Embedding 处理，存入 ChromaDB 用于语义检索。

#### 3.6.1 语义边界切分

合同有明确的条款结构，不能用固定 token 数粗暴切分。**按合同语义单元切分**：

| 切分层级 | 分隔符 | 示例 |
|----------|--------|------|
| 第一级 | 条款标题（第X条/第X章） | "第三条 付款方式" |
| 第二级 | 款（1./2./(一)/(二)） | "1. 首付款：合同签订后..." |
| 第三级 | 自然段（\n\n） | 段落内完整语义 |
| 兜底 | 512 token 强制截断 | 防止单条过长 |


合同中的**金额、日期、百分比**必须归一化后再向量化，否则检索精度大幅下降。

##### 金额归一化

```
原始文本：
  "合同总金额为人民币伍拾万元整（¥500,000.00）"
  "合同价款：50万元"
  "合同金额：500000元"
  "总价：500,000.00 CNY"

归一化后（附加到chunk文本末尾）：
  "合同总金额为人民币伍拾万元整（¥500,000.00） [AMOUNT:500000.00CNY]"
  "合同价款：50万元 [AMOUNT:500000.00CNY]"
  "合同金额：500000元 [AMOUNT:500000.00CNY]"
  "总价：500,000.00 CNY [AMOUNT:500000.00CNY]"
```

**归一化规则**：
```python
import re

def normalize_amount(text: str) -> str:
    """金额归一化：统一转为 [AMOUNT:数字CNY] 标记"""
    
    # 中文大写金额 → 数字
    cn_num_map = {"零":0,"一":1,"二":2,"三":3,"四":4,"五":5,
                  "六":6,"七":7,"八":8,"九":9,"十":10,"百":100,
                  "千":1000,"万":10000,"拾":"十","佰":"百","仟":"千"}
    
    # 模式1: ¥500,000.00 / 500,000.00元
    pattern1 = r'(¥\s*[\d,]+\.?\d*|[\d,]+\.?\d*\s*元)'
    text = re.sub(pattern1, lambda m: f'{m.group()} [AMOUNT:{extract_number(m.group())}CNY]', text)
    
    # 模式2: 人民币伍拾万元整
    pattern2 = r'人民币[零一二三四五六七八九十百千万亿拾佰仟]+元整'
    text = re.sub(pattern2, lambda m: f'{m.group()} [AMOUNT:{cn_to_num(m.group())}CNY]', text)
    
    return text
```

#### 3.6.5 表格内容处理

合同中的表格（付款计划表、软件清单、交付清单等）不能简单按文本切分，需特殊处理。

**策略**：表格 → 两种格式双写

```
┌──────────────────────────────────────────────┐
│ 原始表格（Word/PDF中的表格）                    │
│ ┌──────┬──────┬──────────┬─────────┐         │
│ │ 阶段  │ 比例  │ 金额(元)   │ 触发条件  │         │
│ ├──────┼──────┼──────────┼─────────┤         │
│ │ 首付款 │ 30%  │ 150,000  │ 签约后   │         │
│ │ 中期款 │ 40%  │ 200,000  │ 验收后   │         │
│ │ 尾款   │ 30%  │ 150,000  │ 试运行后 │         │
│ └──────┴──────┴──────────┴─────────┘         │
└──────────────────────┬───────────────────────┘
                       │
          ┌────────────┴────────────┐
          ▼                         ▼
┌──────────────────┐    ┌──────────────────────┐
│ Markdown 表格     │    │ JSON 结构化           │
│ (向量检索用)      │    │ (精确查询用)          │
├──────────────────┤    ├──────────────────────┤

│                  │    │      "比例":"30%",      │
│ → 存入ChromaDB    │    │      "金额":150000,...},│

│                  │    │   ]                     │
│                  │    │ }                       │
```

```python
def process_table(table_data: list, table_context: str) -> dict:
    """
    处理合同表格，返回双格式
    
    Args:
        table_data: 表格二维数组
        table_context: 表格上下文（如"第三条 付款方式"）
    
    Returns:
        {
            "markdown": "| 阶段 | 比例 | ...",   # 用于向量化
            "json": {...},                       # 存入metadata
            "metadata_tags": ["付款", "表格", "金额"]
        }
    """
    # 1. 生成 Markdown 格式
    md_lines = []
    # 表头
    md_lines.append("| " + " | ".join(table_data[0]) + " |")
    md_lines.append("|" + "|".join(["------"] * len(table_data[0])) + "|")
    # 数据行
    for row in table_data[1:]:
        md_lines.append("| " + " | ".join(str(c) for c in row) + " |")
    
    markdown_table = "\n".join(md_lines)
    
    # 2. 生成 JSON 格式
    headers = table_data[0]
    json_rows = []
    for row in table_data[1:]:
        json_rows.append({headers[i]: row[i] for i in range(len(headers))})
    
    table_json = {
        "context": table_context,
        "headers": headers,
        "rows": json_rows
    }
    
    # 3. 对 Markdown 表格也做金额/日期归一化
    markdown_table = normalize_amount(markdown_table)
    markdown_table = normalize_date(markdown_table)
    
    return {
        "markdown": markdown_table,
        "json": table_json,
        "metadata_tags": ["表格"] + infer_tags_from_headers(headers)
    }
```

#### 3.6.6 完整 Embedding 流水线

```
合同文本 + 解析JSON
  │
  ├─ 1. 表格检测与提取
  │     ├─ 检测 Word 表格 (python-docx table)
  │     ├─ 检测 PDF 表格 (PyMuPDF + 启发式规则)
  │     └─ 表格 → MD + JSON 双写
  │
  ├─ 2. 特殊内容归一化
  │     ├─ 金额归一化 → [AMOUNT:数字CNY]
  │     ├─ 日期归一化 → [DATE:YYYY-MM-DD] / [DURATION:X_days]
  │     └─ 百分比归一化 → [PCT:X%]
  │
  ├─ 3. 语义边界切分
  │     ├─ 按合同条款层级切分
  │     └─ 保留条款标题作为 section_title
  │
  ├─ 4. 元数据标注
  │     ├─ 文档级元数据（从 Agent1 的 JSON 获取）
  │     ├─ Chunk 级元数据（条款类型、层级、标签）
  │     └─ 特殊内容标记（has_amount/has_date/has_percentage/has_table）
  │
  ├─ 5. 向量化写入 ChromaDB
  │     ├─ Collection: contracts_chunks（条款级向量）
  │     └─ Collection: contracts_tables（表格向量，单独collection）
  │
  └─ 6. 输出统计
        {chunks: 156, tables: 23, amounts_normalized: 45, dates_normalized: 32}
```

### 3.7 向量存储模块 (`vector_store.py`)

**功能**：管理 ChromaDB，双 Collection 存储，用于 Agent 语义检索。

**ChromaDB 配置**：
- 持久化路径：`./data/chroma/`
- Embedding 维度：768（bge-base-zh-v1.5）
- 距离度量：cosine

**双 Collection 设计**：

| Collection | 存储内容 | 用途 |
|------------|----------|------|
| `contracts_chunks` | 合同条款分块（含归一化标记） | 语义检索 |
| `contracts_tables` | 表格 Markdown 文本 | 表格内容检索 |

**核心操作**：

- `search_with_filter(query, metadata_filter)` — 按元数据过滤检索
- `get_stats()` — 获取统计

### 3.11 上传触发解析流水线（三写流程）

```
POST /api/admin/upload  (批量文件)
  │
  ├─ 1. 保存原始文件 → ./data/contracts/
  ├─ 2. 调用 document_reader 提取文本（含表格检测）
  ├─ 3. 调用 contract_parser_agent 提取结构化 JSON
  │
  ├─ 4. 调用 insert_contract tool → MySQL 事务性写入:
  │     ├─ INSERT INTO contracts (主体/日期/金额/全文)
  │     ├─ INSERT INTO contract_installments (分期付款)
  │     ├─ INSERT INTO contract_products (软件产品+硬件产品，product_type区分)
  │     └─ INSERT INTO contract_key_clauses (关键条款)
  │
  ├─ 5. ChromaDB 写入（向量数据）:
  │     ├─ 5.1 表格处理 → MD + JSON 双写
  │     ├─ 5.2 金额/日期/百分比归一化
  │     ├─ 5.3 语义边界切分（按合同条款层级）
  │     ├─ 5.4 元数据标注（文档级+Chunk级）
  │     ├─ 5.5 Embedding 向量化
  │     └─ 5.6 双 Collection 写入:
  │           ├─ contracts_chunks（条款向量 + 元数据）
  │           └─ contracts_tables（表格向量）
  │
  └─ 6. 返回 {
        filename, status, contract_id,
        stats: {mysql_ok, chroma_chunks: N, chroma_tables: N, amounts: N, dates: N}
      }
```

> **上传时的 Tool 调用关系**：`/api/admin/upload` 后台接口直接调用 `insert_contract` tool（不走 Agent ReAct 推理），`insert_contract` 内部调用 L3 数据执行层的 `dao_insert_contract()` 完成事务性写入。同理，`/api/admin/delete/{id}` 调用 `delete_contract` tool → `dao_delete_contract()` + ChromaDB 同步清理。

## 4. 数据流设计

### 4.1 文档导入流程（三写）

```
用户上传文件 → 保存到 ./data/contracts/
    → 文档解析器提取文本
    → LLM 解析为标准化 JSON
    → 【写1】MySQL: contracts + installments + products + key_clauses
    → 【写2】文本归一化 → 语义切分 → Embedding向量化 → ChromaDB contracts_chunks
    → 【写3】表格提取 → MD格式化 → Embedding向量化 → ChromaDB contracts_tables
    → 返回导入状态
```

### 4.2 问答流程

```
用户输入问题
    → Agent ReAct 推理（Thought → Action → Observation 循环）
    → 根据需要调用工具:
        ├─ 结构化查询工具 (MySQL):
        │   ├─ query_con_sum (金额汇总，支持 party_like 模糊条件)
        │   ├─ query_con_count (合同统计，支持 party_like 模糊条件)
        │   ├─ search_con_text (关键词搜索+公司简称模糊匹配)
        │   ├─ get_con_details (合同详情)
        │   └─ list_contracts (合同列表)
        └─ 向量检索工具 (ChromaDB):
            └─ search_vector (语义相似度检索)
    → 综合所有 Observation
    → LLM 生成最终答案（SSE流式）
    → 返回 {answer, sources}
```

**公司简称模糊匹配子流程**：

```
用户输入含公司简称（如"华为""腾讯"）
    → Agent Thought: 识别到公司简称
    → Action: search_con_text("华为")
        → dao_search_text 执行:
           ① party_a LIKE '%华为%' OR party_b LIKE '%华为%' → 匹配完整公司名
           ② full_text LIKE '%华为%' → 匹配合同原文
           ③ 合并结果，附带金额汇总
    → Observation: 返回完整公司名列表 + 金额
    → 如果无结果: Agent 利用 LLM 知识推理简称对应的完整公司名，重试搜索
    → 如果仍无结果: 告知用户未找到相关合同
    → Agent 根据匹配结果直接回答，无需额外调用 query_con_sum
```

4. **公司简称模糊匹配（MySQL）**："华为的合同总金额是多少？" → `search_con_text("华为")` 一步返回匹配到的完整公司名+金额汇总
5. **交付标准（ChromaDB）**："软件开发合同的验收标准是什么？" → `search_vector`
6. **期限查询（MySQL+ChromaDB）**："保密协议的保密期限是多久？" → `get_con_details` + `search_vector`
7. **条款对比（ChromaDB）**："外包合同和采购合同的违约金条款有什么不同？" → `search_vector`
8. **公司简称+多维筛选（MySQL）**："2024年腾讯的合同有几份？" → `search_con_text("腾讯")` 获取完整公司名 → `query_con_count({"party_like":"腾讯","sign_year":"2024"})`
